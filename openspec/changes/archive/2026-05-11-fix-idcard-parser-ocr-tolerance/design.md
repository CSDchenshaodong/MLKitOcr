## Context

当前 `IdCardFrontParser` 在真机上的表现与单元测试差距很大。问题出在解析器对 ML Kit OCR 输出的假设过于理想化：

- 假设标签和值总是在同一水平行
- 假设标签总是被完整识别
- 假设所有文字块的 bounding box 整齐排列
- 同行判定使用固定 24px 容差

真实 OCR 输出的特点：
- bounding box 受拍摄角度、光照、分辨率影响，波动很大
- 中文标签常被拆分为多个 OCR 行或与值混在一起
- 字符识别存在一定错误率（5%-15%）
- 不同拍摄距离下文字大小差异显著

本项目为 Android 离线 OCR 应用，JDK 21 + Kotlin，无 Jetpack Compose。

## Goals / Non-Goals

**Goals:**
- 让解析器在真实拍摄条件下能稳定提取姓名、性别、民族、出生日期、地址、身份证号码
- 保持解析器纯 Kotlin、无外部依赖，可继续用 JUnit 单元测试
- 与 `ReviewActivity` 和 `MainActivity` 的接口兼容性最大化（仅增加 warnings 字段）
- 尽量不降低单元测试中原有理想场景的精度

**Non-Goals:**
- 不做图片预处理优化（裁剪、去噪等属于 imagepreprocess 模块）
- 不引入 OpenCV 或任何 C++ 视觉库
- 不改变 ML Kit OCR 引擎参数配置
- 不修改 Review 界面或 CameraX 相关代码
- 不追求 100% 无编辑自动化

## Decisions

### 决策 1：锚点匹配策略从精确匹配改为前缀 + 编辑距离双层匹配

**方案**：不再仅用 `contains`，引入两级匹配：
- Level 1（前缀匹配）：将 OCR 行文本按字符拆开，检测是否以标签前缀开头（如 `姓名` 匹配 `姓名张三`、`姓名`）
- Level 2（编辑距离）：对短标签（2-4字）计算 OCR 文本片段与标签间的编辑距离，阈值 ≤1 视为匹配

**备选**：用字符重叠度（最长公共子序列比例）替代编辑距离。两者效果接近，编辑距离实现更简单。

**选择理由**：编辑距离能处理单字符误识别（`性刃`→`性别` 编辑距离=1），前缀匹配能处理标签值合并场景。

### 决策 2：同行容差从固定 24px 改为锚点行高的 0.6 倍

**方案**：`isSameRow` 改为：
```kotlin
val tolerance = (anchor.bounds.bottom - anchor.bounds.top) * 0.6f
abs(candidate.centerY - anchor.centerY) <= tolerance
```

**选择理由**：行高已在 30-80px 范围波动，0.6 倍行高对小字体（18px）、大字体（48px）都能合理覆盖，无需手动调参。

### 决策 3：值提取增加多方向搜索

**方案**：`extractRightValue` 改为：
1. 先尝试同行右侧（现有逻辑，但用动态容差）
2. 若同行无结果，搜索正下方（top >= anchor.bottom, left 接近 anchor.left）
3. 若仍无结果，搜索右下区域（top >= anchor.bottom, left >= anchor.right）

**选择理由**：身份证排版中标签-值的布局以「同行右侧」为主、「下方另起行」为辅。先右后下的搜索顺序符合实际版式。

### 决策 4：回退策略采用「锚点基准 + 正则兜底」分层降级

**方案**：
```
hasKnownAnchor() == true  → 用锚点正常提取
hasKnownAnchor() == false → 尝试正则提取 ID 号 → 以此为基准推算其他字段区域 → 正则提取日期
                         → 仍失败 → 纯正则从全量文本匹配 → LOW 置信度
```

回退层级：
1. **部分锚点可用**：以已匹配锚点为基准，推算未匹配字段的搜索区域
2. **仅 ID 号锚点可用**：以 ID 号行位置推算地址、出生日期区域
3. **无任何锚点**：正则匹配 18 位 ID 号 + YYYY-MM-DD 日期 → 标记 LOW

**选择理由**：身份证号码的正则匹配命中率最高（18位数字/X，有校验位可验证），是最可靠的回归基点。以此向上反推其他字段比盲搜更可靠。

### 决策 5：地址提取改为「锚点下区域收集 + 终止边界」

**方案**：移除原有的 `left >= addressAnchor.right` 硬约束，改为：
- 收集 `addressAnchor.top` 到 `idNumberAnchor.top`（或图像底部）之间的所有行
- 排除锚点行本身
- 排除 `公民身份号码` 相关行
- 按 `top` 排序合并

**选择理由**：地址区域范围远大于其他字段，不应该受锚点 left/right 约束。真实身份证上地址文本的 left 可能小于或大于 `住址` 锚点的 right。

### 决策 6：`ParseResult` 增加 `warnings: List<String>`

**方案**：`IdCardFrontParseResult` 添加 `warnings: List<String> = emptyList()`，记录非致命问题，如哪些字段使用了回退策略。

**选择理由**：`ReviewActivity` 可据此在对应字段旁显示提示图标，不改变已有 UI 结构（可后续利用）。

## Risks / Trade-offs

- [编辑距离误匹配]：编辑距离 ≤1 的短文本容易误匹配（如 `户籍` 与 `住址` 编辑距离=2 不会误匹配，但 `性别` 和 `性名` 会匹配）。→ 限制编辑距离仅在候选文本长度相近时使用（长度差 ≤2）
- [回退策略精确度下降]：无锚点回退的精确度必然低于锚点驱动。→ 明确标记 LOW 置信度，Review 界面强制用户核对
- [地址合并可能引入噪声]：扩大了地址收集范围可能混入非地址文本。→ 以 ID 号行作为终止边界 + 排除已知标签行
